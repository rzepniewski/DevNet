    Schema Reference¶

    Complete schema reference for Testbed as Code, covering both the YAMALE runtime schema and the JSON Schema for IDE support.

    Overview¶

    TBaC uses two schema files that mirror the same data model for different purposes:

    File Format Purpose
    schema/schema.yaml YAMALE Runtime validation via Ansible (testbed_schema_validate plugin)
    schema/schema.json JSON Schema Draft 7 IDE-time assistance — autocompletion, hover docs, inline validation

    Both files must be kept in sync when adding new fields to the data model.

    IDE Schema Support (VS Code)¶

    The JSON Schema (schema/schema.json) provides real-time assistance while authoring *.tbac.yaml files in VS Code. No Ansible run is required — feedback is instant as you type.

    Benefits¶

    Feature Description
    Autocompletion Press Ctrl+Space inside any mapping to see all valid property names
    Hover documentation Hover over any key to see its description and type information
    Inline validation Missing required fields and wrong types are flagged immediately with squiggles
    Enum hints Fields like deploy_from and type show allowed values in the dropdown
    Required field enforcement Required fields that are absent are highlighted before you run Ansible

    Setup¶

    Step 1 — Install the Red Hat YAML extension

    Search for redhat.vscode-yaml in the VS Code Extensions panel (Cmd+Shift+X) and install it.

    Step 2 — Confirm the schema association

    The association is already configured in .vscode/settings.json in the repo:

    {
  "yaml.schemas": {
    "./schema/schema.json": "**/*.tbac.yaml"
  }
}

    No further configuration is needed.

    Step 3 — Reload VS Code (first time only)

    Open the Command Palette (Cmd+Shift+P) and run Developer: Reload Window.

    Verifying It Is Working¶

    1. Open any *.tbac.yaml file — try data/testbed.tbac.yaml or any file under examples/

    2. What you should see immediately:

      • Hover over any key → tooltip shows the field description and type
      • Type a new key inside testbed: → autocomplete dropdown shows all valid properties
      • A required field that is missing → yellow/red squiggle with a validation message
      • An invalid value (wrong type, disallowed enum value) → inline error

    3. Check the status bar — at the bottom of the VS Code window, the right side shows the active schema name (e.g. tbac) when a matched file is open. If it shows no schema, the extension may need a reload.

    Note

    The JSON Schema is used solely for IDE assistance. Runtime validation continues to use the YAMALE schema via the testbed_schema_validate Ansible action plugin. When adding a new field, update both schema/schema.yaml and schema/schema.json.

    YAMALE Schema (Runtime Validation)¶

    Root Schema¶

    testbed: include('testbed', required=True)

    Field Types¶

    Basic Types¶

    Type Description Example
    str() String value name: "tbac_vm01"
    int() Integer value cpu: 4
    bool() Boolean value enabled: true
    enum() Enumerated values type: enum('shared', 'dmz')
    ip(version=4) IPv4 address ip: "192.168.1.10"
    any() Any type value: any()

    Complex Types¶

    Type Description Example
    list() List of items dns: list(str())
    include() Reference to schema vm: include('vm_def')

    Schema Definitions Summary¶

    The following table summarises all schema-defined types with their key fields and constraints:

    Root and Hypervisor¶

    Schema Type Key Fields Notes
    testbed name, description, type(shared/dmz), pyats, hypervisor(required), compute, network_services, application_services, images Root object
    pyats_config create_testbed_file(bool), output_directory(str) pyATS integration — both fields optional
    hypervisor vcenter[], standalone_esxi[], openstack, proxmox, nutanix Only vCenter and ESXi implemented
    vcenter host(req), port, username(req), password(req), validate_certs(req), datacenters[], ova_import_async_timeout, ova_import_poll_interval Bare hostname only
    standalone_esxi name, host(req), port, username(req), password(req), validate_certs, datastore(req), vm_names[], ova_import_async_timeout, ova_import_poll_interval Direct ESXi connection
    datacenter name(req), non-cluster, clusters[] Under vCenter
    cluster name(req), datastore, vm_names[], esxi_hosts[] Under datacenter

    Compute¶

    Schema Type Key Fields Notes
    vm name(req), deploy_from(req: template/ova), folder, ova_image_name, template, hardware(req), network(req), os_config(req), services, applications VM definition
    template_config name, guest_id, guestinfo_cloud_init Template deployment
    hardware cpu(req, min=1), memory_mb(req, min=256), disk_gb(req, min=10) Hardware specs
    network_interface name(req), portgroup(req), ip(req, IPv4), netmask, prefix(1-31), gateway, dvswitch_name NIC config
    os_config hostname(req, regex), dns[], domain, timezone, users[], docker_compose_dir, docker_network_name, docker_ubuntu_codename, proxy OS customisation
    os_user username(req), password, sudo, allow_password_auth, ssh_public_key_file, ssh_private_key_file Additional users
    vm_proxy server(req), port(1-65535), username, password, no_proxy[] HTTP/HTTPS proxy
    vm_services ntp, dns, dhcp, http, tftp, tacacs, ldap, snmp Per-VM network service enables
    vm_applications netbox Per-VM application service enables

    Application Services¶

    Schema Type Key Fields Notes
    application_services netbox Application service stack configurations
    netbox_service secret_key(req, min=50), db_password(req), netbox_image, postgres_image, redis_image, db_name, db_user, http_port, allowed_hosts[], base_url, superuser, plugins[], custom_config DCIM/IPAM — three-container stack
    netbox_superuser username(req), password(req), email Superuser created on first deploy
    netbox_plugin name(req), config Plugin to enable in Netbox
