    Frequently Asked Questions¶

    Common questions and answers about Testbed as Code.

    General Questions¶

    What is Testbed as Code?¶

    Testbed as Code is an Infrastructure as Code (IaC) solution for automating the deployment and configuration of virtual machine testbeds on VMware infrastructure. It follows Network as Code principles and uses Ansible for automation.

    Is Testbed as Code free to use?¶

    Yes, Testbed as Code is open-source and licensed under the MIT License. It's free for both personal and commercial use.

    What infrastructure does it support?¶

    Currently supports:

    • ✅ VMware vCenter Server (7.0+)
    • ✅ VMware ESXi (7.0+, standalone)
    • 🚧 OpenStack (planned)
    • 🚧 Proxmox (planned)
    • 🚧 Nutanix AHV (planned)

    Installation & Setup¶

    What are the prerequisites?¶

    • Python 3.12 or higher
    • Ansible 2.19 or higher
    • Access to vCenter or ESXi
    • macOS or Linux (Windows not supported)
    • OVA files or VM templates

    See Prerequisites for details.

    Why is Windows not supported?¶

    The project uses POSIX-specific features and Ansible modules that are not compatible with Windows. You can use WSL2 (Windows Subsystem for Linux) as a workaround.

    Should I use UV or pyenv for Python management?¶

    Both work great!

    • UV (recommended): 10-100x faster than pip, built-in Python management, modern tooling
    • pyenv: Mature, widely adopted, auto-activation support

    See Installation Guide for both options.

    Configuration¶

    What's the difference between defaults/ and data/?¶

    • defaults/: Reusable default values shared across multiple VMs or environments
    • data/: Environment-specific configurations that override defaults

    The merge system intelligently combines both, with data values taking precedence.

    Learn more →

    How do I handle secrets and passwords?¶

    Use Ansible Vault to encrypt all sensitive data:

    # Encrypt a password
ansible-vault encrypt_string 'your_password' --name 'password'

# Create vault password file
echo 'vault_password' > .vault_pass
chmod 600 .vault_pass

    Can I deploy multiple VMs at once?¶

    Yes! Simply define multiple VMs in your configuration:

    compute:
  vms:
    - name: "tbac_vm01"
      # ... config ...
    - name: "tbac_vm02"
      # ... config ...
    - name: "tbac_vm03"
      # ... config ...

    Multi-VM Example →

    What image formats are supported?¶

    • OVA files: VMware Open Virtual Appliance format
    • VM Templates: Existing templates in vCenter
    • ISO files: For custom installations (advanced)

    OVA files can be sourced from:

    • Local files on the Ansible control node
    • Artifactory or private registries (with authentication)
    • Remote servers via SCP or SFTP (requires sshpass on the control node)

    Downloaded OVAs are cached locally in the images/ directory to avoid re-downloading.

    Deployment & Operations¶

    How long does deployment take?¶

    Deployment time depends on several factors:

    • VM deployment: 5-15 minutes per VM (depends on OVA size)
    • Network services: 2-5 minutes
    • Validation: 1-2 minutes

    A typical 3-VM environment with services: 20-30 minutes

    Can I update an existing testbed?¶

    Yes! The system supports incremental updates:

    1. Modify your configuration files
    2. Run validation: ansible-playbook playbooks/tbac.yaml --tags validate
    3. Review changes in .tbac/changes.diff
    4. Deploy updates: ansible-playbook playbooks/tbac.yaml --tags deploy

    What if validation fails?¶

    Check the error output for specific issues:

    • Schema errors: Fix YAML structure or required fields
    • Semantic errors: Address business logic violations
    • Connectivity errors: Verify network access and credentials

    Validation Guide →

    How do I roll back changes?¶

    1. Git history: Revert configuration to previous commit
    2. Model tracking: Review .tbac/previous.tbac.yaml for last working state
    3. Remove role: Use the remove role with safety flags to clean up

    Rollback Considerations

    Always test rollback procedures in non-production environments first.

    Advanced Topics¶

    Can I add custom network services?¶

    Yes! The architecture is extensible. To add a new service:

    1. Create service configuration in data model
    2. Add YAMALE schema validation
    3. Create deployment tasks in roles/deploy/tasks/service_deployment/
    4. Add unit tests

    Plugin Development Guide →

    Does it work with CI/CD pipelines?¶

    Absolutely! Testbed as Code is designed for automation:

    • Validation in CI: Run schema/semantic validation in pull requests
    • Automated deployment: Deploy testbeds on schedule or trigger
    • Testing integration: Use pyATS testbed files for automated testing

    Example GitHub Actions workflow is included.

    What about multi-environment deployments?¶

    Use the configurable merge directories feature:

    # Development environment
ansible-playbook playbooks/tbac.yaml -e "data_dir=data-dev"

# Staging environment
ansible-playbook playbooks/tbac.yaml -e "data_dir=data-staging"

# Production environment
ansible-playbook playbooks/tbac.yaml -e "data_dir=data-prod"

    Can I integrate with existing automation?¶

    Yes! Testbed as Code generates pyATS testbed files automatically:

    # Generated at out/pyats_testbed.yaml
testbed:
  name: testbed_as_code
devices:
  WEB_SERVER_01:
    type: server
    os: linux
    connections:
      ssh:
        ip: 192.168.1.10

    Use this file with your pyATS test scripts.

    Troubleshooting¶

    Common Error: "Unable to connect to vCenter"¶

    Check:

    1. Network connectivity: ping vcenter.lab.local
    2. Credentials are correct and not expired
    3. vCenter API is accessible: curl -k https://vcenter.lab.local
    4. Firewall allows HTTPS (port 443)

    Common Error: "Schema validation failed"¶

    Solutions:

    1. Review error message for specific field issues
    2. Check YAML syntax: python -c "import yaml; yaml.safe_load(open('data/testbed.tbac.yaml'))"
    3. Validate against schema: python scripts/validate_schema.py
    4. Refer to Schema Reference

    Common Error: "OVA file not found"¶

    Check:

    1. File path is correct and absolute
    2. File permissions allow read access
    3. For remote URLs, network connectivity exists
    4. For Artifactory, authentication is configured

    Common Error: "Unable to resolve image reference"¶

    Cause: A VM references an image name that doesn't exist in the image catalog.

    Fix: Add the image to the images: section of your data file, or correct the ova_image_name field on the VM to match an existing catalog entry.

    Common Error: "VM already exists"¶

    Cause: A VM with the same name is already deployed in vCenter/ESXi.

    Fix: Remove the existing VM from vCenter first, or use a different name in your compute configuration.

    Common Error: "Insufficient resources"¶

    Cause: The target ESXi host does not have enough CPU or memory to power on the VM.

    Fix: Free resources on the host, reduce the VM's cpu or memory_mb settings, or deploy to a different host with available capacity.

    Where can I find logs?¶

    • Ansible logs: ansible.log (configured in ansible.cfg)
    • Model tracking: .tbac/run_map.yaml for change tracking
    • Generated configs: .tbac/current.tbac.yaml
    • Change diffs: .tbac/changes.diff
