    SNMP Service¶

    LibreNMS network monitoring using official librenms/librenms Docker image. Provides SNMP polling, trap reception, device discovery, and alerting.

    Official LibreNMS Documentation

    For in-depth details beyond this Testbed as Code reference, see the official LibreNMS docs:

    Topic URL
    Alerting Introduction https://docs.librenms.org/Alerting/
    Alert Rules (syntax, options, examples) https://docs.librenms.org/Alerting/Rules/
    Alert Templates https://docs.librenms.org/Alerting/Templates/
    Alert Transports (mail, Slack, Microsoft Teams, Cisco Webex Teams…) https://docs.librenms.org/Alerting/Transports/
    Alert Macros https://docs.librenms.org/Alerting/Macros/
    Alert Testing https://docs.librenms.org/Alerting/Testing/
    Fast Ping Checking https://docs.librenms.org/Extensions/Fast-Ping-Check/
    SNMP Trap Handler https://docs.librenms.org/Extensions/SNMP-Trap-Handler/
    Dispatcher Service https://docs.librenms.org/Extensions/Dispatcher-Service/
    Docker Installation https://docs.librenms.org/Installation/Docker/
    Configuration https://docs.librenms.org/Support/Configuration/
    API Reference https://docs.librenms.org/API/
    Alert Rules Collection (community rules) https://docs.librenms.org/Alerting/Rules/#alert-rules-collection

    Multi-container architecture

    LibreNMS uses multiple containers: main app, database (MariaDB), cache (Redis), dispatcher, and snmptrapd. All orchestrated via Docker Compose.

    Configuration approach

    Uses three-tier config: Docker environment (TZ, PUID/PGID), LibreNMS environment variables (librenms.env), and application config (librenms-config.yaml).

    Supported Images¶

    Image Supported Versions Notes
    librenms/librenms 26.2.0 (default) Official LibreNMS with all features
    mariadb 10.11 (default) Database backend
    redis 7.2-alpine (default) Cache and session storage

    Basic SNMP Configuration¶

    network_services:
  snmp:
    image_name: "librenms/librenms"
    image_version: "26.2.0"
    db_password: "secure_password_here"
    db_root_password: "secure_root_password"
    web_password: "admin_password_here"
    snmp_community: "private"
    monitored_devices:
      - hostname: "192.168.1.1"
        community: "private"
        snmp_version: "v2c"

    Complete SNMP Example¶

    network_services:
  snmp:
    service_name: "tbac_librenms"
    image_name: "librenms/librenms"
    image_version: "26.2.0"
    db_image_version: "10.11"
    redis_image: "redis:7.2-alpine"
    db_password: "YourStrongPassword123!"
    db_root_password: "YourRootPassword456!"
    web_user: "admin"
    web_password: "YourAdminPassword789!"
    timezone: "America/New_York"
    poller_threads: 16
    discovery_threads: 16
    poller_frequency: 300
    snmp_community: "private"
    snmp_version: "v2c"
    web_port: 8000
    trap_port: 162
    fast_ping: true
    fast_ping_interval: 60
    alert_default_only: false            # false = route alerts per rule's transports list
    alert_transports:
      - name: "Email Ops Team"
        type: "mail"
        is_default: true
        config:
          email: "ops@example.com"
          email_backend: smtp
          email_smtp_host: smtp.example.com
          email_smtp_port: 25
          email_from: '"LibreNMS" <librenms@example.com>'
      - name: "Slack Alerts"
        type: "slack"
        config:
          url: "https://hooks.slack.com/services/YOUR/WEBHOOK/URL"
    alert_rules:
      - name: "Port Down"
        severity: critical
        conditions:
          - field: ports.ifOperStatus
            operator: "="
            value: "down"
          - field: ports.ifAdminStatus
            operator: "="
            value: "up"
        transports: ["Email Ops Team", "Slack Alerts"]
      - name: "Device Down"
        severity: critical
        conditions:
          - field: devices.status
            operator: "="
            value: "0"
        transports: ["Email Ops Team"]
    monitored_devices:
      - hostname: "192.168.1.1"
        community: "private"
        snmp_version: "v2c"
        display_name: "Core Router"
      - hostname: "192.168.1.2"
        snmp_version: "v3"
        snmpv3_authname: "snmpuser"
        snmpv3_authpass: "authpassword123"
        snmpv3_authalgo: "SHA"
        snmpv3_cryptopass: "privacypassword456"
        snmpv3_cryptoalgo: "AES"
        # Note: snmpv3_authlevel is auto-detected (authPriv in this case)
        # based on which credentials are provided - no need to specify it

    SNMP Field Reference¶

    Schema-required fields with defaults

    Many fields marked Yes in the schema are provided automatically by defaults/defaults.tbac.yaml. If you use the defaults file, you only need to specify the three password fields (db_password, db_root_password, web_password) and any environment-specific overrides.

    Field Type Default Required Description
    service_name string tbac_librenms No Service/directory name. Affects container names (<service_name>, <service_name>_db, <service_name>_dispatcher) and host directory path (<docker_compose_dir>/<service_name>/)
    image_name string librenms/librenms Yes Docker image for LibreNMS
    image_version string 26.2.0 Yes LibreNMS version tag
    db_image_version string 10.11 Yes MariaDB image version (e.g. 10.11)
    redis_image string redis:7.2-alpine Yes Redis image (e.g. redis:7.2-alpine)
    db_password string - Yes Database password — must be set; no default. Use ansible-vault
    db_root_password string - Yes MySQL root password — must be set; no default. Use ansible-vault
    web_password string - Yes Web UI admin password — must be set; no default. Use ansible-vault
    db_host string db Yes Database container name/hostname
    db_port integer 3306 Yes Database port (1-65535)
    db_name string librenms Yes Database name
    db_user string librenms Yes Database username
    web_user string admin Yes Web UI admin username
    timezone string UTC Yes Server timezone (e.g., America/New_York)
    poller_threads integer 16 Yes Number of parallel poller workers (1-64)
    discovery_threads integer 16 Yes Number of parallel discovery workers (1-64)
    poller_frequency integer 300 Yes Polling and alert evaluation interval in seconds (60-300). Lower values = faster alerts but more CPU/SNMP load
    snmp_community string private Yes Default SNMP v1/v2c community string
    snmp_version string v2c Yes Default SNMP version (v1, v2c, v3)
    snmp_timeout integer 5 Yes SNMP timeout in seconds (min: 1)
    snmp_retries integer 3 Yes SNMP retry attempts (min: 0)
    auto_discovery boolean true Yes Enable XDP-based auto-discovery
    web_port integer 8000 Yes Web interface HTTP port (1-65535)
    trap_port integer 162 Yes SNMP trap receiver UDP port (1-65535)
    fast_ping boolean true Yes Enable fast ICMP ping checks for quicker device down/up detection. Testbed as Code automatically configures this in the dispatcher. See Fast Ping Checking
    fast_ping_interval integer 300 Yes Fast ping interval in seconds (15-300). Lower values = faster detection but higher load. See Fast Ping Checking
    base_url string (auto-generated) No Custom base URL (e.g. https://librenms.example.com). Auto-generates http://<vm-ip>:<web_port> if omitted. Required for CSRF protection when using a reverse proxy
    enable_syslog boolean false No Enable syslog-ng receiver sidecar
    syslog_port integer 514 No Syslog UDP port (if enabled)
    alert_default_only boolean true No When true, alerts are sent only via transports with is_default: true. When false, alerts use per-rule transport lists. Does NOT disable alerting — to stop notifications, remove alert_transports.
    alert_transports list [] No Alert delivery methods (mail, slack, microsoft_teams, cisco_webex_teams, webhook)
    alert_rules list [] No Alert rules that trigger notifications (see Alert Rule Fields)
    monitored_devices list [] No List of devices to monitor via SNMP

    Mail Transport Config Fields¶

    These fields go inside the config: block of a type: mail transport. They map directly to LibreNMS Mail Transport settings.

    Data Config Key LibreNMS DB Key Type Default Required Description
    mail-contact mail-contact string email No Contact type: email (specified address), sysContact (from device SNMP), owners (device owners), role (LibreNMS user role)
    email email string - When mail-contact: email Recipient email address
    role role string "" When mail-contact: role LibreNMS user role name to send alerts to
    bcc bcc boolean false No Send as BCC instead of TO
    attach-graph attach-graph boolean true No Embed graph images in alert emails (requires @signedGraphTag in template)
    email_backend - string mail No Email backend: smtp, sendmail, or mail
    email_smtp_host - string localhost When smtp SMTP relay server hostname
    email_smtp_port - integer 25 No SMTP server port
    email_smtp_auth - boolean false No Enable SMTP authentication
    email_smtp_username - string - When auth SMTP username (use ansible-vault)
    email_smtp_password - string - When auth SMTP password (use ansible-vault)
    email_smtp_secure - string "" No SMTP encryption: tls, ssl, or "" (none)
    email_from - string LibreNMS@<container> No Sender "From" address

    Transport config vs global email config

    The first five fields (mail-contact, email, role, bcc, attach-graph) are per-transport settings stored in the alert_transports table. The email_* fields are global LibreNMS email settings configured via lnms config:set — Testbed as Code sets these in librenms-config.yaml automatically.

    network_services:
  snmp:
    fast_ping: true
    alert_default_only: false
    alert_transports:
      - name: "Email Ops Team"
        type: "mail"
        is_default: true
        config:
          mail-contact: email           # Contact type: email, sysContact, owners, role
          email: "ops@example.com"      # Recipient address (when mail-contact: email)
          # role: "admin"              # LibreNMS role name (when mail-contact: role)
          bcc: false                    # Send as BCC instead of TO
          attach-graph: true            # Embed graphs in alert email
          # SMTP relay for alert emails (global settings)
          email_backend: smtp
          email_smtp_host: smtp.example.com
          email_smtp_port: 25
          email_from: '"My-LibreNMS" <librenms@example.com>'
          # For authenticated SMTP:
          # email_smtp_auth: true
          # email_smtp_username: "user@example.com"
          # email_smtp_password: "secret"
          # email_smtp_secure: tls

    Alert Transport Types¶

    Supported alert transport types:

    Type Description Required Fields
    mail Email alerts email, email_backend, email_smtp_host (when smtp)
    slack Slack webhook url, channel (optional)
    microsoft_teams Microsoft Teams webhook url, use-json (optional)
    cisco_webex_teams Cisco Webex Teams api-token, room-id, use-markdown (optional)
    webhook Generic HTTP webhook url, method (optional)

    All transport types

    LibreNMS supports 40+ transport types. Testbed as Code automates the five above. For the full list (PagerDuty, Telegram, Discord, etc.) see Alert Transports.

    Alert Transport Fields¶

    Field Type Default Required Description
    name string - Yes Display name for the transport (must be unique). Referenced by alert_rules[].transports[]
    type string - Yes Transport type: mail, slack, microsoft_teams, cisco_webex_teams, webhook
    is_default boolean false No When true, this transport is automatically applied to all alert rules — even rules that don't explicitly list it in transports[]. Set this on your primary notification channel so no alerts are missed
    config map - Yes Type-specific configuration (see transport type table above for required keys)

    Default transport behavior

    Setting is_default: true on a transport means LibreNMS will use it for every alert rule, regardless of whether the rule's transports list includes it. This is useful for ensuring a primary email or chat channel always receives all alerts. You can still add additional non-default transports to specific rules via the transports field.

    Cisco Webex Teams Transport Config Fields¶

    These fields go inside the config: block of a type: cisco_webex_teams transport. They map directly to LibreNMS Cisco Spark Transport settings.

    Data Config Key LibreNMS DB Key Type Default Required Description
    api-token api-token string - Yes Webex Bot API token. Create at developer.webex.com/my-apps → Create a Bot
    room-id room-id string - Yes Webex space (room) ID. Get via GET https://webexapis.com/v1/rooms
    use-markdown use-markdown boolean false No Send alert messages using Markdown formatting 
    alert_transports:
  - name: "Cisco Webex Teams Alerts"
    type: "cisco_webex_teams"
    is_default: true
    config:
      api-token: "NjE2YzY5..."          # Bot access token
      room-id: "Y2lzY29zcGFyay..."      # Room ID from API
      use-markdown: false                # Use Markdown formatting

    Bot must be a space member

    The bot must be added as a member of the Webex space to post messages. Search the bot's username in the space's People panel to add it.

    Slack Transport Config Fields¶

    These fields go inside the config: block of a type: slack transport. They map to LibreNMS Slack Transport settings.

    Data Config Key LibreNMS DB Key Type Default Required Description
    url slack-url string - Yes Slack Incoming Webhook URL
    channel slack-channel string "" No Override channel (e.g. #alerts). Leave empty to use webhook default

    Slack Author

    Testbed as Code automatically sets the slack-author field to "LibreNMS" in the database. This is not configurable.

    alert_transports:
  - name: "Slack Alerts"
    type: "slack"
    config:
      url: "https://hooks.slack.com/services/T00000/B00000/XXXXXXX"
      channel: "#network-alerts"       # Optional: override webhook default channel

    Microsoft Teams Transport Config Fields¶

    These fields go inside the config: block of a type: microsoft_teams transport. They map to LibreNMS Microsoft Teams Transport settings.

    Data Config Key LibreNMS DB Key Type Default Required Description
    url msteam-url string - Yes Microsoft Teams Incoming Webhook URL
    use-json use-json boolean false No Send alert as JSON MessageCard instead of Markdown. Your alert template must be valid MessageCard JSON

    Testbed as Code uses user-friendly type names that map to LibreNMS internal transport types:

    Testbed as Code type Value LibreNMS Internal Type LibreNMS PHP Class DB Config Keys
    mail mail Mail.php mail-contact, email, role, bcc, attach-graph

    Hyphen vs underscore in config keys

    Config keys with hyphens (e.g. mail-contact, api-token, room-id) can also be written with underscores (e.g. mail_contact, api_token, room_id). Both forms are accepted in your data YAML. Testbed as Code templates handle both transparently.

    Alert Rules¶

    Alert rules define when to fire notifications. Each rule has conditions evaluated against LibreNMS database fields, and a list of transports to notify. Rules are transport-agnostic — the same rule can notify email, Cisco Webex Teams, Microsoft Teams, and Slack simultaneously.

    Official reference

    See Alert Rules — Options for the full list of rule options, and Alert Rules — Examples for community examples.

    alert_rules:
  - name: "Port Down"
    severity: critical
    conditions:
      - field: ports.ifOperStatus
        operator: "="
        value: "down"
      - field: ports.ifAdminStatus
        operator: "="
        value: "up"
    transports: ["Email Ops Team", "Cisco Webex Teams NOC"]

    Alert Rule Fields¶

    Field Type Default Required Description
    name string - Yes Unique rule name
    severity string critical No critical, warning, or ok
    condition string AND No Logical operator: AND or OR
    conditions list - Yes List of match conditions (see below)
    transports list [] No Transport names to notify (must match alert_transports[].name). Transports marked is_default: true are automatically added to all rules
    count integer 1 No Max alerts per event before stopping. 1 = alert once then wait for recovery (recommended). -1 = unlimited re-alerts every interval until condition clears
    delay string 0m No Delay before first alert (e.g. 5m). Useful to avoid alerts for transient issues
    interval string 0m No Re-alert interval until max count is reached (e.g. 5m). Use 0m when count: 1 (no re-alerting)
    mute boolean false No Mute transport notifications (alert still visible in Web UI)
    invert boolean false No Invert rule match — alert when condition is not met instead of when it is
    recovery boolean true No Send recovery notification when condition clears

    One-time alerting (recommended)

    Use count: 1 with interval: 0m to receive one email when a device/port goes down and one recovery email when it comes back up. Setting count: -1 causes continuous re-alerts every interval — avoid this unless you explicitly want repeated notifications.

    See LibreNMS Alert Rules — Options for official documentation on these fields.

    Condition Fields¶

    Field Type Required Description
    field string Yes Database field (e.g. ports.ifOperStatus, devices.status)
    operator string Yes Comparison: =, !=, >, <, >=, <=, like, not like, regex, starts, ends, in, not in, between, is null, is not null
    value string Yes Value to match against

    Available entities and operators

    The field uses the format table.column from the LibreNMS database. Common tables: devices, ports, processors, mempools, storage, syslog. See Alerting Rules — Syntax for all supported operators and Alerting Entities for available fields.

    Docker interface auto-exclusion

    When any condition uses ports.ifOperStatus, Testbed as Code automatically appends exclusions for Docker virtual interfaces (docker0, br-*, veth*) to prevent false Port Down alerts. No data-file changes needed.

    Common Alert Rules¶

    Rule Conditions Description
    Port Down ports.ifOperStatus = down AND ports.ifAdminStatus = up Admin-enabled port went operationally down
    Device Down devices.status = 0 Device is unreachable
    High CPU processors.processor_usage > 80 CPU utilization above 80%
    High Memory mempools.mempool_perc > 90 Memory utilization above 90%
    Storage Full storage.storage_perc > 95 Disk usage above 95%

    Monitored Device Fields¶

    Field Type Default Required Description
    hostname string - Yes Device hostname or IP address
    community string (inherits) For v1/v2c SNMP community string (default: inherits from snmp_community)
    snmp_version string (inherits) No SNMP version: v1, v2c, v3 (default: inherits from snmp_version)
    snmp_port integer 161 No SNMP port (1-65535)
    display_name string - No Custom display name in web UI
    snmpv3_authlevel string - No SNMPv3 security level: noAuthNoPriv, authNoPriv, or authPriv. Optional — auto-inferred by Rule 303 from which credentials are present. Only specify to override inference
    snmpv3_authname string - For v3 auth SNMPv3 username (security name)
    snmpv3_authpass string - For v3 auth Authentication password (use ansible-vault)
    snmpv3_authalgo string - For v3 auth Auth algorithm: MD5, SHA, SHA-224, SHA-256, SHA-384, or SHA-512 (default: SHA; MD5 is deprecated)
    snmpv3_cryptopass string - For authPriv Privacy/encryption password (use ansible-vault)
    snmpv3_cryptoalgo string - For authPriv Privacy algorithm: DES or AES

    Device Addition Process¶

    When Testbed as Code adds monitored devices to LibreNMS, it automatically performs a three-step process for each device:

    1. Device Addition: Adds the device to LibreNMS with the specified SNMP credentials
    2. Initial Discovery: Runs LibreNMS discovery to find ports, CPUs, storage, sensors, and OS type. Without discovery, the ports table remains empty and SNMP traps (linkDown/linkUp) cannot be matched to interfaces
    3. Initial Poll: Collects metrics for everything discovery found. Must run after discovery so polling has interfaces/sensors to collect data for

    This ensures devices are immediately operational with complete monitoring data. The entire process runs automatically during deployment.

    Manual device operations

    After deployment, you can manually trigger discovery and polling: 

    # SSH to the LibreNMS VM
docker exec -u librenms tbac_librenms lnms device:discover <hostname>
docker exec -u librenms tbac_librenms lnms device:poll <hostname>

    Container Architecture¶

    LibreNMS deployment consists of multiple containers:

    Container Name Purpose Ports
    <service_name> Main LibreNMS web interface and core services <web_port>:8000 (default: 8000)
    <service_name>_db MariaDB database for LibreNMS data Internal only
    <service_name>_redis Redis cache for sessions and queues Internal only
    <service_name>_dispatcher Polling and discovery orchestration Internal only
    <service_name>_snmptrapd SNMP trap receiver <trap_port>:162 (TCP/UDP, default: 162)
    <service_name>_syslog Syslog-ng receiver (optional) <syslog_port>:514 (TCP/UDP, if enabled)

    Automatic configuration

    Testbed as Code automatically: - Creates the admin user with specified credentials - Configures fast ping checks in the dispatcher - Sets up alert transports and rules - Adds monitored devices with discovery and initial poll - Configures SMTP settings from mail transport config

    Validation Rules¶

    Rule Description
    302 Docker image validation — librenms/librenms is validated against the whitelist with version pinning
    303 SNMP/LibreNMS service configuration validation — credential strength (blocks common weak passwords), cisco-password restriction (passwords containing "cisco" in any case are rejected by LibreNMS), SNMPv3 credential completeness, alert transport type-specific required fields, alert rule transport reference integrity

    Security Best Practices¶

    Production Security

    Always use Ansible Vault for sensitive fields: - db_password - db_root_password - web_password - snmpv3_authpass - snmpv3_cryptopass - email_smtp_password - Alert transport API tokens

    # Encrypt sensitive values
ansible-vault encrypt_string 'YourSecretPassword' --name 'db_password'

    SNMP Community Strings

    • Never use default communities (public, private) in production
    • Use strong, unique community strings per device or device group
    • Consider SNMPv3 for devices that support it (better security)

    Alert Configuration

    • Set is_default: true on your primary notification channel (email) to ensure no alerts are missed
    • Use count: 1 and interval: 0m for one-time alerting (recommended)
    • Add delay: 5m to rules that may have transient issues (e.g., high CPU)
    • Use the recovery: true flag to get notified when issues clear

    Troubleshooting¶

    Device not being monitored: - Check SNMP is enabled on the target device - Verify community string or SNMPv3 credentials are correct - Test SNMP manually: snmpwalk -v2c -c <community> <device-ip> - Check LibreNMS logs: docker logs tbac_librenms

    Alerts not being sent: - Verify transport configuration in LibreNMS web UI (Alerts → Alert Transports) - Check alert rules are enabled (Alerts → Alert Rules) - Test transport: LibreNMS UI → Alert Transports → Test - Review alert log: LibreNMS UI → Alerts → Alert Log

    Web interface not accessible: - Verify containers are running: docker ps | grep librenms - Check web port is open: curl http://localhost:8000 - Review container logs: docker logs tbac_librenms
