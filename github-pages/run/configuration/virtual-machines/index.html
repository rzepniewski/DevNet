    Virtual Machine Configuration¶

    Define VM specifications, hardware, networking, and deployment methods.

    Overview¶

    VMs are defined in the compute.vms section with complete hardware, network, and OS configurations.

    Default VM Credentials¶

    VMs deployed from the TBaC base OVA image ship with a built-in OS user:

    Field Value
    Username ubuntu
    Password cisco.123

    Cannot Be Changed via Data Model

    The default ubuntu user and its password are baked into the OVA image and cannot be changed through the data model. They are intended for initial console/SSH access only.

    To define additional users with custom passwords, SSH keys, and sudo privileges, use the os_config.users block (see OS Configuration below).

    First-Boot-Only Configuration (Cloud-Init)¶

    The following settings are applied only at initial VM deployment via cloud-init and are never modified on subsequent runs of the deploy role:

    • IP address (network.interfaces[].ip)
    • Netmask (network.interfaces[].netmask / prefix)
    • Default gateway (network.interfaces[].gateway)
    • Hostname (os_config.hostname)
    • DNS servers (os_config.dns)
    • Users (os_config.users)

    By Design

    TBaC intentionally avoids overwriting network or OS configuration on VMs that are already deployed. Re-running ansible-playbook playbooks/tbac.yaml --tags deploy against an existing VM will not update its IP address, gateway, netmask, or any other cloud-init setting, even if the values have been changed in the data model or directly on the VM.

    To apply changes, the VM must be destroyed and redeployed.

    Basic VM Definition¶

    compute:
  vms:
    - name: "tbac_vm01"
      deploy_from: "ova"
      ova_image_name: "tbac-base-ubuntu"
      folder: "my-vms"              # Optional: Folder name only (not full path)
      hardware:
        cpu: 2
        memory_mb: 4096
        disk_gb: 40
      network:
        interfaces:
          - name: "ens192"
            portgroup: "VM Network"
            ip: "192.168.1.10"
            netmask: "255.255.255.0"
            gateway: "192.168.1.1"
      os_config:
        hostname: "tbac_vm01"
        dns: ["8.8.8.8"]

    vCenter Folder Configuration¶

    IMPORTANT: The folder field specifies only the folder name, not the full vCenter path.

    Correct Usage¶

    vms:
  - name: "tbac_vm01"
    folder: "my_folder"                  # ✅ CORRECT - Simple folder name
    # folder: "/my_folder"               # ✅ CORRECT - Leading slash is allowed
    # folder: "parent/child/grandchild"  # ✅ CORRECT - Nested path under vm/

    Incorrect Usage¶

    vms:
  - name: "tbac_vm01"
    # folder: "/UK-DC/vm/my_folder"      # ❌ WRONG - Includes datacenter prefix
    # folder: "UK-DC/vm/my_folder"       # ❌ WRONG - Includes datacenter/vm prefix

    How Path Construction Works¶

    The deployment system automatically constructs the full vCenter path:

    Format: {datacenter_name}/vm/{folder_value}

    Example: - Datacenter: UK-DC - Folder configured: my_folder - Full path created: UK-DC/vm/my_folder

    Nested folders: - Folder configured: team/project/test - Full path created: UK-DC/vm/team/project/test

    Automatic Folder Creation¶

    • The system automatically creates folders if they don't exist
    • Parent folders are created recursively as needed
    • Folder creation is idempotent (safe to run multiple times)
    • Supports special characters (e.g., email addresses in folder names)

    Deployment Methods¶

    Deploy from OVA¶

    - name: "tbac_vm01"
  deploy_from: "ova"
  ova_image_name: "tbac-base-ubuntu"  # References images catalog

    Validation Requirements¶

    When using deploy_from: "ova", the following validations are enforced:

    1. ova_image_name is required - Must reference an image from the images catalog
    2. Referenced image must exist - The image name must match an entry in the images section
    3. Image must have a source - Each image must have an artifact object and/or a remote object
    4. Artifact must have path - If artifact is defined, the artifact.path field is required
    5. Remote must have required fields - If remote is defined, protocol, ip, username, password, and path_location are required (Rule 801)

    Example validation error messages:

    ❌ VM 'web-vm' has deploy_from='ova' but missing 'ova_image_name' to reference an image from the catalog
❌ VM 'web-vm' references ova_image_name='tbac-base-ubuntu' but no image with that name exists in the images catalog
❌ Image 'tbac-base-ubuntu' is missing 'artifact' configuration
❌ Image 'tbac-base-ubuntu' is missing 'artifact.path'
❌ Image 'tbac-base-ubuntu' remote source missing required field 'password'

    See Image Catalog Configuration for complete image catalog documentation.

    Deploy from Template (experimental)¶

    - name: "tbac_vm02"
  deploy_from: "template"
  template:
    name: "ubuntu-24.04-template"
    guest_id: "ubuntu64Guest"
    guestinfo_cloud_init: true

    Hardware Configuration¶

    hardware:
  cpu: 4          # Number of vCPUs (min: 1, required)
  memory_mb: 8192 # Memory in MB (min: 256, required)
  disk_gb: 100    # Disk size in GB (min: 10, required)

    Network Configuration¶

    Single Interface¶

    network:
  interfaces:
    - name: "ens192"
      portgroup: "VLAN-100"
      ip: "10.0.100.10"
      netmask: "255.255.255.0"
      gateway: "10.0.100.1"

    Multiple Interfaces¶

    network:
  interfaces:
    - name: "ens192"        # Management
      portgroup: "MGMT"
      ip: "192.168.1.10"
      netmask: "255.255.255.0"
      gateway: "192.168.1.1"

    - name: "ens224"        # Data
      portgroup: "DATA"
      ip: "10.0.0.10"
      netmask: "255.255.255.0"

    - name: "ens256"        # Storage
      portgroup: "STORAGE"
      ip: "172.16.0.10"
      netmask: "255.255.255.0"

    OS Configuration¶

    os_config:
  hostname: "tbac_vm01"
  domain: "lab.local"
  timezone: "America/New_York"
  dns: ["8.8.8.8", "8.8.4.4"]
  docker_compose_dir: "/opt/net-services"    # From defaults
  docker_network_name: "net_services"         # From defaults
  docker_ubuntu_codename: "noble"             # From defaults

  # Additional user accounts (on top of default 'ubuntu' user)
  users:
    - username: "tbac"
      password: "CHANGE_ME_USE_VAULT"  # Use ansible-vault encrypt_string
      sudo: true
      allow_password_auth: true
      ssh_public_key_file: "~/.ssh/id_ed25519.pub"
      ssh_private_key_file: "~/.ssh/id_ed25519"

    Encrypt Passwords with Ansible Vault

    ansible-vault encrypt_string 'CHANGE_ME_USE_VAULT' --name 'password'

    os_config.users Field Reference¶

    Field Type Required Description
    username string Yes Login username for the additional user
    password string No User password (use ansible-vault encrypt_string to encrypt)
    sudo bool No Grant sudo privileges (true/false)
    allow_password_auth bool No Allow SSH password authentication (true/false)
    ssh_public_key_file string No Path to SSH public key file for key-based auth
    ssh_private_key_file string No Path to SSH private key file (used by Ansible for connections)

    Note

    The os_config.users block defines additional users created via cloud-init on first boot. The default ubuntu / cisco.123 user from the OVA image remains available regardless of what is configured here.

    Proxy Configuration¶

    Configure HTTP/HTTPS proxy settings for VMs deployed in environments that require a proxy for outbound internet access. The proxy block is optional — if omitted, no proxy settings are applied.

    Minimal Example¶

    os_config:
  hostname: "tbac_vm01"
  proxy:
    server: "proxy.example.com"
    port: 8080

    Full Example (Authenticated Proxy with Bypass List)¶

    os_config:
  hostname: "tbac_vm01"
  proxy:
    server: "proxy.example.com"       # Bare hostname or IP — no http:// prefix
    port: 8080                        # Proxy port (required)
    username: "tbac"                  # For authenticated proxies
    password: !vault |                # Always encrypt with Ansible Vault
      $ANSIBLE_VAULT;1.1;AES256
      ...
    no_proxy:                         # Addresses that bypass the proxy
      - "localhost"
      - "127.0.0.1"
      - ".internal.example.com"
      - "10.0.0.0/8"

    os_config.proxy Field Reference¶

    Field Type Required Description
    server string Yes Proxy server hostname or IP. Do not include http:// or https:// — the deploy task builds the full URL.
    port integer No Proxy port number (1-65535). Validated by Rule 150 when proxy block is defined.
    username string No Username for authenticated proxy access. Must be paired with password.
    password string No Password for authenticated proxy access. Must be paired with username. Use ansible-vault encrypt_string.
    no_proxy list(string) No Addresses, domains, or CIDRs that bypass the proxy.

    Server Format

    The server field must be a bare hostname or IP address. Values like http://proxy.example.com will fail semantic validation (Rule 150). The deployment task automatically constructs the full http://[user:pass@]server:port URL.

    Defaults

    If port is omitted, the deploy task uses 80 as a fallback via | default(80). Specify port explicitly if your proxy runs on a non-standard port.

    What Gets Configured¶

    When the proxy block is defined, the deploy role writes two files on the target VM:

    1. /etc/environment — System-wide http_proxy, https_proxy, HTTP_PROXY, HTTPS_PROXY (and no_proxy/NO_PROXY if defined)
    2. /etc/apt/apt.conf.d/95proxy — APT-specific Acquire::http::Proxy and Acquire::https::Proxy

    Running Proxy Deployment¶

    # Deploy only proxy configuration
ansible-playbook playbooks/tbac.yaml --tags deploy_proxy

# Deploy proxy as part of base OS setup
ansible-playbook playbooks/tbac.yaml --tags deploy_base

    Service Assignment¶

    Assign network services to VMs. Each service is independently enabled using the services block.

    - name: "tbac_vm01"
  # ... hardware and network config ...
  services:          # Network services (see network_services config)
    dns:
      enable: true
    dhcp:
      enable: true
    ntp:
      enable: true
    tftp:
      enable: true
    http:
      enable: true
    ldap:
      enable: true
    tacacs:
      enable: true
    snmp:
      enable: true
  applications:      # Application services (see application_services config)
    netbox:
      enable: true

    services vs applications

    Use services to enable network services (DNS, DHCP, NTP, TFTP, HTTP, LDAP, TACACS+, SNMP/LibreNMS). Use applications to enable application services (Netbox). The two blocks are separate because application services involve multi-container stacks with database migrations, while network services are single-container deployments.
