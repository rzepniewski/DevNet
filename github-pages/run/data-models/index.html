    Data Model¶

    Complete reference for the Testbed as Code data model structure.

    Overview¶

    The Testbed as Code data model provides a declarative, YAML-based approach to defining virtual machine testbed infrastructure. The model follows Network as Code principles with a hierarchical structure that separates infrastructure concerns into logical sections.

    Data Model Structure¶

    The data model is organized into five main configuration areas:

    Hypervisor¶

    Infrastructure deployment targets including vCenter servers, ESXi hosts, and future platform support (OpenStack, Proxmox, Nutanix).

    Compute¶

    Virtual machine definitions including hardware specifications, network configuration, and OS customization.

    Network Services¶

    Network infrastructure services (DNS, DHCP, NTP, TFTP, HTTP, LDAP, TACACS+, SNMP/LibreNMS) with Docker-based deployment configuration.

    Application Services¶

    Application deployments for services like Netbox, Splunk, and custom applications (future).

    Images¶

    Centralized image catalog for OVA and ISO files with authentication and version management.

    Root Structure¶

    The root testbed object contains metadata and references to the five main sections:

    testbed:
  name: str                    # Testbed identifier (optional)
  description: str             # Purpose description (optional)
  type: enum('shared', 'dmz')  # Testbed type (optional)
  pyats: {...}                 # Optional: pyATS integration settings

  hypervisor: {...}            # Required: Infrastructure targets
  compute: {...}               # Optional: VM definitions
  network_services: {...}      # Optional: Network services
  application_services: {...}  # Optional: Applications
  images: [...]                # Optional: Image catalog

    Root Fields¶

    Field Type Required Description
    name string No Testbed identifier for reference
    description string No Human-readable description of testbed purpose
    type enum No Testbed type: shared (multi-user) or dmz (isolated)
    pyats object No pyATS integration settings (see pyATS reference)
    hypervisor object Yes Infrastructure deployment targets
    compute object No Virtual machine definitions
    network_services object No DNS, DHCP, NTP, TFTP, HTTP, LDAP, TACACS+, SNMP/LibreNMS service configurations
    application_services object No Application service definitions
    images list No Centralized image catalog

    Data Model Validation¶

    All configurations are validated against the YAMALE schema before deployment:

    Validation Flow¶

    graph LR
    A[YAML Config] --> B[Schema Validation]
    B --> C[Semantic Validation]
    C --> D[Deployment]

    style A fill:#049fd9,color:#fff
    style B fill:#0a60ff,color:#fff
    style C fill:#0a60ff,color:#fff
    style D fill:#00c853,color:#fff
    1. Schema Validation: YAMALE checks structure, types, required fields
    2. Semantic Validation: Python rules verify business logic and relationships
    3. Deployment: Configuration applied to infrastructure

    Complete Example¶

    Here's a complete testbed configuration showing all sections:

    testbed:
  name: "production-lab"
  description: "Production testing environment"
  type: "shared"

  hypervisor:
    vcenter:
      - host: "vcenter.lab.local"
        username: "admin@vsphere.local"
        password: !vault |encrypted|
        validate_certs: false
        datacenters:
          - name: "Lab-DC"
            clusters:
              - name: "Cluster1"
                datastore: "shared-storage"
                vm_names: ["tbac_vm01", "tbac_vm02"]

  images:
    - name: "tbac-base-ubuntu"
      type: "ova"
      version: "1.0.0"
      artifact:
        location: "artifactory"
        path: "NetworkServices/TBAC/tbac_2404-1.ova"
        url: "https://artifactory.example.com/generic-local"
        auth:
          type: "bearer_token"
          token: !vault |encrypted|

  compute:
    vms:
      - name: "tbac_vm01"
        deploy_from: "ova"
        ova_image_name: "tbac-base-ubuntu"
        folder: "/production"
        hardware:
          cpu: 4
          memory_mb: 8192
          disk_gb: 100
        network:
          interfaces:
            - name: "ens192"
              portgroup: "VLAN-100"
              ip: "192.168.1.10"
              netmask: "255.255.255.0"
              gateway: "192.168.1.1"
        os_config:
          hostname: "tbac_vm01"
          dns: ["8.8.8.8", "8.8.4.4"]
          domain: "lab.local"
          timezone: "America/New_York"
          users:
            - username: "admin"
              password: "CHANGE_ME_USE_VAULT"
              sudo: true
        services:
          dns:
            enable: true

  network_services:
    dns:
      upstream_servers: ["8.8.8.8", "8.8.4.4"]
      a_records:
        - name: "web.local."
          type: "A"
          value: "192.168.1.10"

    dhcp:
      interfaces: ["ens192"]
      subnets:
        - cidr: "192.168.1.0/24"
          range_start: "192.168.1.100"
          range_end: "192.168.1.200"
          router: "192.168.1.1"
          dns_servers: ["192.168.1.10"]

    ntp:
      allow_networks: ["192.168.1.0/24"]
      upstream_servers:
        - "0.pool.ntp.org"
        - name: "time.google.com"
          options: "iburst prefer"

  application_services: {}

    Best Practices¶

    1. Use the image catalog - Define images once, reference everywhere
    2. Leverage defaults - Use the merge system for shared configurations
    3. Validate early - Run schema validation during development
    4. Document your configs - Add comments to complex configurations
    5. Version control - Track all configuration changes in Git
